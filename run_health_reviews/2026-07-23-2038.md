**Root cause** (from job logs, `publish.py:443`):
```
postgrest.exceptions.APIError: {'message': 'invalid input syntax for type integer: "6.5"', 'code': '22P02', ...}
```
`transactions.json`'s 2026-07-22 PLD BUY carries `research_confidence: 6.5` — the
**first non-integer confidence value ever recorded** (all ~40 prior trades back to
June are whole numbers 5–7). The Supabase `trades.research_confidence` column is
typed `integer`; Postgres accepts an implicit cast from a whole-number float (`6.0`)
but rejects one with a fractional part (`6.5`). `publish.py`'s only except-branch
around the `trades` upsert (lines 444–453) handles the unrelated `PGRST204`
broker_order_id schema-cache-lag case and re-raises everything else — so this
exception is uncaught and crashes the script.

**Blast radius (verified from `publish.py`'s statement order, lines 368→409→443):**
- `portfolio_snapshots` upsert (incl. today's `close_value`) — **succeeds**, runs
  before the crash point. Confirmed by the log line
  `📊 Snapshot published: value=$506.95 return=+1.39% spy=+1.26%` printing
  before the traceback. The website's headline value/return chart is **not**

- `positions` upsert — **succeeds**, also runs before the crash point.

  EOG BUY, PLD BUY) have never reached the website's `trades` table.
- `quant_scores` upsert — **never reached** (script exits before this line runs).
- This is **not transient** — every future run re-sends the full non-dry-run
  `transactions.json` history, so the same 6.5-valued row will break the upsert
  **every single day going forward** until either the code or the schema changes.
  It will not self-heal.
